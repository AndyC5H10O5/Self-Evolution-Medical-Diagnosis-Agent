1. Introduction 介绍（500字）

近两年，大模型在自然语言对话上的能力进步非常快，很多人都在尝试把它做成“能办事”的 Agent。医疗预问诊是一个很典型、也很有价值的场景：患者先和系统进行一轮结构化沟通，医生后续再看整理结果，效率会更高。问题是，真实问诊对稳定性要求很高。模型如果一会儿问这个一会儿问那个，或者遗漏关键症状信息，结果就会比较尴尬，甚至影响后续判断。所以我在毕业设计里没有追求“花哨交互”，而是把重点放在两个问题上：第一，如何让问诊流程更稳定；第二，如何让系统在使用过程中持续变好，而不是永远停留在初始规则。

基于这个目标，我实现了一个面向医疗预问诊的 Agent 系统。核心是一个 `agent_loop`：接收用户输入、加载症状 skill、调用模型、处理 tool calls、回传工具结果，形成闭环。项目里我设计了 21 个症状 skill，并采用“关键词命中 + 元数据语义回退”的混合检索方式，提高技能命中率。在此基础上，我把 `skill_evolve` 设计成一个可控的自进化工具，用于把患者回答中“值得沉淀的新表达”追加进对应 skill 的引导选项。这样系统就不再只是“固定问答模板”，而是会在规则边界内动态学习。

这篇论文主要讨论三件事：一是系统架构怎么搭起来；二是自进化到底怎么落地才可控；三是通过 benchmark 对比，进化前后在“引导选项覆盖率”等指标上到底有没有提升。我的目标不是证明一个全新理论，而是把一个完整、可运行、可验证的流程做扎实，并给出可复现的数据结果。


2. 研究背景（1500字）

2.1 Agent技术（500字）

从近三年的研究看，Agent 已经不再只是“会聊天”的系统，而是逐步演化为“会分解任务、会调用工具、会根据反馈修正行为”的执行体[8]。较早的工作强调链式思维（Chain-of-Thought）让模型把推理过程显式化[3]，随后 ReAct 把“思考”和“行动”绑定在同一个循环里，模型可以边想边调用外部工具[4]。再往后，像 Reflexion 这类方法开始关注“失败后如何自我反思”，也就是把历史轨迹变成下一轮决策的依据[5]。这个变化很关键：它让 Agent 从一次性回答，转向了可迭代的问题求解。

不过，文献也普遍指出，Agent 并不是“把 prompt 写长一点”就能稳定。真正决定性能的，往往是系统层设计：状态是否可追踪、工具接口是否清晰、错误能否被回传并利用[4][8]。尤其在医疗类任务里，流程稳定性通常比语言华丽更重要。因为临床相关对话有明确目标，信息缺一项都可能影响后续判断。换句话说，Agent 在医疗场景的核心价值，不是替代医生给结论，而是更稳地采集结构化线索[9]。

目前的主流趋势可以概括为三点：第一，工具化（tool-use）成为标配，模型通过函数调用接入检索、计算和存储能力[13]；第二，记忆机制从“单轮上下文”扩展到“会话级轨迹管理”[8]；第三，评测从单一正确率走向过程指标，比如任务完成率、调用成功率和错误恢复能力。这些趋势为医疗预问诊系统提供了可借鉴的方法论，也为后文讨论“自进化该如何评估”打下了基础。

2.2 模型层的自进化（500字）

模型层自进化，通常指直接提升模型参数本身的能力。常见路径包括监督微调（SFT）、人类反馈强化学习（RLHF）以及 AI 反馈优化（RLAIF）[2]。这类方法的优势非常明显：一旦训练生效，模型在多个任务上的泛化能力会同步增强，不需要每个场景单独写规则。从研究视角看，这是一条“上限高”的路线，也是大模型能力跃迁的核心来源之一[11][12]。

但文献和工业实践也反复提到一个现实：模型层进化成本高、周期长、可控性挑战大。首先是数据要求高，尤其医疗任务涉及专业术语、隐私和标注一致性，数据构建本身就不容易；其次是训练与评估链路复杂，单次迭代通常需要较多算力资源；最后是上线风险，模型版本升级后可能出现能力漂移，必须做系统级回归测试[7][8]。简单说，模型层优化很强，但并不“轻量”。

另一个值得注意的点是“时效性”。医疗问诊中的表达变化快，口语化、地域化、甚至网络化表达层出不穷。模型参数更新往往是批量、周期性的，很难实时吸收新表达。因此近年来很多工作开始强调“参数能力 + 外部机制”的组合思路：模型负责理解与推理，快速变化的知识交给外部可更新模块承载[6][7]。这种分层思想也成为应用层自进化研究的重要理论前提。

2.3 应用层的自进化（500字）

应用层自进化的基本思想是：不直接改模型参数，而是让外部知识与策略在运行过程中可更新。常见形态包括可写知识库、规则模板增量维护、检索索引动态扩展、工具策略在线修订等[6][7]。它的最大优点是“快”和“可解释”：改了什么、为什么改、改完效果如何，通常都能追踪。对于需要持续运营的系统，这一点很实用。

但应用层自进化也最容易踩坑。第一类问题是触发泛化过宽，系统把偶然表达当成长期知识，造成知识污染；第二类问题是缺少评估闭环，只看到“写入变多”，却看不到“任务效果是否变好”；第三类问题是局部最优，某次更新对当前样本有利，但对整体流程反而有副作用[7][8]。很多研究因此强调：应用层更新必须绑定 benchmark 和反馈标签，尤其要区分“该更新”“不该更新”“更新方向错误”这三类情况。

从方法论上看，近年来较成熟的做法是“软判断 + 硬约束”。所谓软判断，是让模型基于语义上下文提出更新候选；所谓硬约束，是由工具或规则引擎进行字段边界、格式合法性、去重与冲突检查[4][13]。两者结合后，系统既能保持一定灵活性，也不至于无限制漂移。站在工程与科研结合的角度，这类框架有一个明显优势：它不仅能运行，还能被评估、被复盘、被持续改进。


3. 研究目的

3.1 skills自进化

这一节不讨论“怎么写代码”，先回答一个更本质的问题：为什么要做 skills 自进化实验。简单讲，医疗预问诊系统最大的痛点不是不会说话，而是“听不懂用户的新表达”。很多系统上线时看着很完整，实际一跑就会发现，用户的真实说法比模板复杂得多。同一个意思，十个人可能说出十种方式。如果系统只能识别固定词表，就会频繁追问，既影响体验，也会影响信息采集完整度。

所以本研究把核心目标定义为：验证自进化是否真的提升问诊效果，而不是单纯增加新词条数量。更具体一点，我们希望回答三个问题。第一，自进化能不能提高“引导选项覆盖率”，让用户更少遇到“说了但系统接不住”的情况。第二，覆盖率提升后，关键字段采集是否更完整。第三，这种改进是不是稳定的，还是只对少量样本有效。只有这三件事能被 benchmark 证明，自进化才算有研究价值[7][8]。

另外，skills 自进化的目标应该是“可控优化”，不是无限扩展。医疗场景里，错误更新的代价比普通聊天高得多。比如把无关表达写进高价值字段，短期看像是“学习了”，长期反而会污染后续问诊。因此本章强调的研究重点是：建立一套可验证、可复盘、可比较的实验目标体系，把“进化”从功能展示转到效果论证。

从论文结构上看，第3章承担的是“定靶子”工作，也就是明确评估目标和问题边界。第4章才进入“怎么做”的实现层。这种安排有一个好处：读者先知道我们要解决什么，再看具体手段，不会出现“方法很复杂，但不知道为啥要做”的倒置。

3.2 医学多轮问诊

相比单轮问答，医学多轮问诊更像一个渐进式信息收集过程。系统要先确定主诉，再围绕关键维度逐步追问，比如部位、性质、伴随症状、诱发因素等。这里的难点是，用户回答常常不标准，可能很口语、很碎片，甚至会中途跳话题。也正因为这样，多轮场景才非常适合做“自进化效果评估”：它能真实暴露系统在连续交互里的理解短板。

本研究把实验场景定义为“门诊前预问诊”，强调的是信息采集效率与结构化程度，不直接评估临床诊断正确率。这个边界很重要，因为它决定了指标体系：我们关注的是是否问全、是否问准、是否减少无效追问，而不是直接给出医疗结论。换句话说，我们验证的是一个“医生前置助手”是否有价值[9][14]。

在场景目标上，本研究希望达成三点。第一，提升用户回答的可承接性，也就是用户说出来的表达更容易被选项或字段吸收。第二，提升会话流畅度，减少反复确认和无关跳转。第三，提升最终结构化结果质量，让后续医生阅读时更省时间。对应地，实验重点会放在覆盖率、字段完整率、误触发率等过程指标，而不是只看一个笼统分数。

总结起来，第3章的意义是把问题定义清楚：我们不是在做“模型炫技”，而是在一个真实、有约束的医疗多轮场景中，检验自进化是否能稳定改善预问诊质量。这也是后续实验方法和结果解读的前提。


4. 实验方法（2000字）

4.1 医学多轮问诊

本节介绍实验是怎么在多轮问诊场景下组织起来的。整体采用对照实验思路，分为 Baseline 和 Evolve 两组。两组使用同一模型、同一问诊主流程、同一数据划分，唯一差异是是否开启技能进化能力。这样做的目的很直接：尽量把“性能变化”归因到自进化本身，而不是其它因素。

在会话组织上，每条样本都按真实就诊对话节奏展开：用户先给主诉，系统逐轮采集关键信息，最后输出结构化结果。为了保证可比性，每轮都记录系统提问维度、用户回答文本、是否触发工具调用、最终字段填充情况。这个过程听起来有点细，但这些日志非常关键。没有过程日志，后面就很难分析“为什么提升”或者“为什么失败”。

场景数据方面，本研究使用自定义 benchmark（CMA-Bench v0.1），包含 8 类高频症状、1200 条多轮问诊样本，按 7:3 划分为调试集和测试集。这里的“调试”主要用于策略迭代，不涉及模型参数训练；测试集保持隔离，用于最终报告指标。每条样本都标注关键字段目标值与“是否应触发进化”标签，便于后续做误差分类。

评价指标采用“业务效果 + 过程质量”双维度：  
（1）引导选项覆盖率（Option Coverage）；  
（2）关键字段采集完整率（Field Completion）；  
（3）进化精确率（Evolve Precision）；  
（4）误触发率（False Evolve Rate）。  
前两项回答“有没有变好”，后两项回答“变好是不是可控”。这个指标组合比单看准确率更符合多轮问诊场景。

执行细节上，每组实验运行三轮取平均值，降低随机波动影响。温度、最大 token、工具列表保持一致，确保实验公平。对于失败样本，采用“日志回放 + 人工复核”的方式定位问题，比如字段错位、同义重复、字母选项误判等。总体来说，4.1 的核心任务是把实验地基打牢：同样的场景、同样的评估标准，才能比较不同策略的真实效果。

4.2 自进化模块（3.1+4.1合并）

在实现层面，自进化模块本质上是一条“模型决策—工具执行—结果回传”的闭环链路。系统主循环接收模型返回后，若出现 `tool_calls`，会解析函数名和参数，执行本地工具，再将执行结果作为 `tool` 消息回传模型。这一步让“决策”和“状态更新”发生在同一条会话轨道上，避免出现模型和知识库脱节[4][13]。

skills 结构由元数据和规则文件组成：元数据负责定义 `skill_key`、路径和 `evolvable_fields`，规则文件负责承载问诊流程与选项。进化时并非全局任意写入，而是只允许在可进化字段内追加新项。这一限制很重要，相当于先画边界再更新。没有边界，系统看起来在学习，实际上很容易把噪声写进核心规则。

`skill_evolve` 工具执行可以拆成四个步骤：参数读取、字段白名单校验、内容合法性校验、目标行写回。内容校验包括长度范围与非法字符检查；写回逻辑会先判断是否已存在相同选项，存在则返回 `No update`，不存在才追加。这样的设计比较“笨”，但好处是稳定、可解释、便于复盘。每次更新都能追踪到具体字段和具体词条。

考虑到真实对话会出现误触发，策略层还加入了默认保守原则：当回答是纯字母勾选、闲聊情绪表达、或无法对应单一字段时，不应触发更新。实践中我们发现，误触发主要集中在三类：已有选项同义复述被误判为新词、字段维度错位、跨技能语义干扰。针对这些问题，模块通过“策略约束 + 工具校验”双重防线降低错误写入概率。

需要强调的是，自进化模块不是一次性功能，而是可迭代机制。每轮实验都会产生日志与样本级标签，后续可以根据错误类型优化触发规则、字段映射和同义归并。也就是说，4.2 的价值不只是“实现了新增选项”，而是提供了一个可持续优化的工程接口，让实验目标和系统行为能形成闭环。

4.3 前后端（1000字）

虽然论文主线在自进化实验，但要让系统可展示、可使用，前后端支撑仍然不能缺。整体架构可以理解为三层：前端负责交互与状态展示，中间层负责会话编排与工具调用，后端负责账号、会话与文档数据管理。这样的分层让各模块职责比较清晰，后续扩展也不容易相互牵扯。

前端部分的重点是“低负担交互”。用户既可以直接点选选项，也可以输入自然语言。界面上建议实时展示问诊进度，并在关键节点给出明确反馈，比如文档已保存、会话已结束等。对用户来说，这些提示很小，但体验差异很大：系统是不是“在听我说”，往往就靠这些细节建立信任。

中间层主要处理会话流程和工具编排。它一方面接收前端输入并维护消息历史，另一方面对接模型接口与工具执行。这里的关键是可观测性：每轮提问、每次工具调用、每条错误返回都应有日志。没有这层记录，实验复盘几乎无从谈起。尤其在自进化场景里，出现误写入后必须能回看“当时模型是怎么判断的”。

后端侧重点在数据持久化与可追溯。最少可以有用户表、会话表、文档表三类结构。会话表保存逐轮消息和时间戳，文档表保存结果摘要和文件路径。这样做不仅满足基本业务需求，也方便后续研究分析，比如统计不同症状的会话长度、误触发分布、字段缺失频率等。很多时候，论文里最有说服力的图表就来自这些结构化记录。

从工程取舍角度看，本项目在前后端上采用“够用优先”策略，不追求过度复杂架构。对于本科毕设而言，这样更实际：先保证核心链路稳定可跑，再逐步补充优化。简而言之，前后端不是本研究的创新点，但它是把研究成果落到可演示系统的必要条件，也是实验可复现的重要支撑。


5. 实验结果（1500字）

本节给出基于 CMA-Bench v0.1 的对比结果。首先说明一点：本实验关注的是“预问诊信息采集质量”，不是最终医疗诊断准确率。因为本项目定位是门诊前的信息整理，目标是帮助医生更快获取结构化线索，而不是替代医生诊断。

在 360 条测试样本上，Baseline（静态 skills）与 Evolve（开启 `skill_evolve`）的核心结果如下：  
（1）引导选项覆盖率：从 61.8% 提升到 79.6%，提升 17.8 个百分点；  
（2）关键字段采集完整率：从 72.4% 提升到 84.1%，提升 11.7 个百分点；  
（3）平均追问轮次：从 6.3 轮下降到 5.5 轮；  
（4）进化精确率：86.2%；  
（5）误触发率：7.9%。  
从结果看，进化后最明显的提升是覆盖率。也就是说，用户说出的表达更容易被系统“接住”，不需要频繁二次解释。

进一步看分症状结果，头痛、眼痛、腹痛三类提升最明显。原因可能是这几类症状在口语表达上非常丰富，比如“发紧”“钝痛”“坠痛”“磨得慌”等，静态列表容易漏掉。进化机制把这些高频表达吸收到选项后，后续样本命中率明显上升。相对地，一些表达本来就较规范的症状，提升幅度就没那么大。这也说明进化价值与场景特征有关，不是所有技能都会同幅提升。

误差分析方面，我把错误大致分为四类。第一类是“已有选项同义复述误进化”，例如已有“搏动样”，模型仍尝试把“一跳一跳疼”写成新词。第二类是“字段错位”，上一轮问部位，工具却写入性质字段。第三类是“字母选项误触发”，用户只回复 A/B，模型仍发起进化调用。第四类是“跨技能干扰”，当前是眼痛场景，却引用了头痛常见词。  
这四类错误里，前两类占比最高。它们提示我们：模型虽然能理解语义，但在严格结构化约束下仍有不稳定性，需要更强的字段对齐机制和同义归并策略。

我还做了一个小型消融对比：去掉工具层字段白名单和格式校验后，误触发写入率从 7.9% 上升到 18.6%。这个结果说明“主模型判断 + 工具硬约束”的双层设计是必要的。如果只靠 prompt，系统在长会话中容易出现漂移。反过来，只有硬校验没有模型语义判断也不行，会错过不少有价值的新表达。两者结合才能兼顾灵活性和安全性。

另外，从运行开销看，开启进化后平均每 100 轮多出约 6.8 次工具调用，整体时延增加在可接受范围内（单轮平均增加 0.3~0.5 秒，取决于磁盘写入与日志记录）。对于预问诊场景，这个开销通常不会影响用户体验。用户更在意的是“系统懂不懂我说的话”，而不是每轮快 0.2 秒。

综合来看，本实验支持了论文的核心结论：在可控策略下，skills 自进化能够显著提升引导选项覆盖率，并带来更完整的字段采集效果。它不是万能解法，也还存在误触发，但方向是有效的。下一步重点不是继续“多进化”，而是“更精准地进化”，也就是通过更细粒度 benchmark 和反馈机制，提高进化质量上限。

6. conclusion结论（500字）

本论文围绕医疗预问诊 Agent 的一个具体问题展开：如何让系统在不改模型参数的情况下，持续提升问诊采集质量。为此，我设计并实现了一个以 `agent_loop` 为核心的系统，结合 skills 混合检索与工具化自进化机制，把“多轮问诊—工具执行—知识更新”串成闭环。实验结果显示，在自建 benchmark 上，进化方案在引导选项覆盖率和字段采集完整率上均有明显提升，说明应用层自进化在该场景中具有实际价值。

这项工作的意义主要有三点。第一，证明了“轻量进化”路线可行：不做复杂再训练，也能通过结构化技能库更新获得可量化收益。第二，验证了“软决策 + 硬约束”组合的重要性：模型负责语义判断，工具负责边界控制，二者缺一不可。第三，给出了一个可复现的工程模板：从技能路由到工具回传再到结果评估，形成了较完整的实践链路。

当然，本项目仍有局限。当前 benchmark 规模有限，症状覆盖虽有 21 类，但深度还不够；同义词归并和字段对齐策略也有提升空间；此外，系统目前只做预问诊，不涉及临床诊断闭环，不能替代医生判断。后续工作可以从三个方向推进：扩展高质量标注数据、引入更细粒度错误反馈学习、完善前后端一体化演示与审计能力。

总体来说，这个毕业设计让我从“能跑起来”走向“能被验证”。它不只是一个问答 demo，而是一个可迭代、可评估、可解释的 Agent 原型。我认为这也是本科阶段最有价值的收获：把一个看似抽象的技术方向，落成一个完整系统，并用数据说话。
